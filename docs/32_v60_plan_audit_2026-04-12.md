# Audit Report: Module 2/3 v6.0 Comprehensive Plan

Date: 2026-04-12  
Auditor role: computational simulation scientist, wet-lab chemistry researcher, physicochemical model reviewer  
Scope: `docs/31_v60_plan.md` and the current EmulSim M2/M3 implementation.

## 1. Executive Verdict

The v6.0 plan is directionally strong. Its central principle, building calibration, uncertainty, and lifetime frameworks before adding more chemistry, is scientifically correct and aligned with how chromatography/process simulation becomes useful in real wet-lab and industrial settings. The plan correctly recognizes that uncalibrated HIC, lectin affinity, lifetime, pore accessibility, and batch variability cannot be made reliable by adding more reagent profiles alone.

The plan is not yet implementation-ready as written. It assumes a v5.9 baseline that is mostly present in code, but not fully verified. Current code has 52 M2 reagent profiles, 9 M2 step types, 10 ACS site types, FMC v2 fields, and an M3 isotherm adapter module. However, direct testing shows the charge-exchange adapter path currently fails because `select_isotherm_from_fmc()` calls `SMAIsotherm(q_max=...)`, while `SMAIsotherm` does not accept `q_max`. The existing M3 tests pass because they do not cover this adapter route. This makes v6.0’s M3-dependent plans less reliable until the v5.9 routing baseline is fixed.

Primary recommendation: proceed with v6.0 only after a v5.9 hardening step that verifies M3 adapter routing, gradient-process-state propagation, UI exposure of all existing step types, and unit consistency. Then implement v6.0 as framework-first alpha releases, starting with calibration and contract corrections, not HIC or lifetime outputs.

## 3. Current M2/M3 System Role

M2’s role is to convert an M1-generated base microsphere into a chemically functional material. It must represent the post-fabrication chemistry that determines whether the microsphere becomes an IEX resin, IMAC resin, affinity resin, HIC resin, catalytic bead, or unsupported intermediate material.

M2 should therefore produce:

- ACS conservation and terminal-state accounting.
- Activation, crosslinking, spacer, coupling, quenching, metal charging, protein pretreatment, and washing histories.
- Functional ligand density on a scientifically valid area basis.
- Material state, including metal loading, residual reagents, protein activity retention, and intermediate groups.
- Confidence, calibration, and validity metadata.
- A `FunctionalMediaContract` that M3 can consume without guessing the chemistry.

M3’s role is to translate that functional material into process performance:

- breakthrough curves,
- elution curves,
- detector signals,
- pressure/flow effects,
- process metrics such as dynamic binding capacity, yield, purity, and lifetime performance.

The current system is structurally close to this goal, but still has critical integration gaps. In particular, chemistry-specific M3 routing and gradient-dependent equilibrium are not yet robust enough to support all v6.0 claims.

## 4. Baseline Assessment

The v6.0 plan states a v5.9 baseline of 52 profiles, 9 step types, 10 ACS types, FMC v2, and M3 isotherm routing. The local implementation mostly matches the count-based baseline:

- 52 `ReagentProfile` entries are present.
- 9 `ModificationStepType` values are present.
- 10 `ACSSiteType` values are present.
- FMC v2-style fields such as `reagent_accessible_area_per_bed_volume`, `ligand_accessible_area_per_bed_volume`, `q_max_lower`, `q_max_upper`, and `m3_support_level` exist.
- `SPACER_ARM`, `METAL_CHARGING`, `PROTEIN_PRETREATMENT`, and `WASHING` backend logic exists.
- A M3 adapter module exists for `binding_model_hint` routing.

But the baseline is not fully reliable:

- The SMA adapter route currently fails.
- The irreversible adsorption adapter is not safely vectorized for LRM array inputs.
- `run_breakthrough()` does not automatically select an isotherm from FMC unless an isotherm is explicitly supplied elsewhere.
- Gradient elution still evaluates a scalar gradient and does not pass time-varying process state into adsorption equilibrium.
- The UI exposes `SPACER_ARM` but not `METAL_CHARGING`, `PROTEIN_PRETREATMENT`, or `WASHING`.
- The proposed new `src/emulsim/uncertainty/` package conflicts with the existing `src/emulsim/uncertainty.py` module name.

These are not blockers to v6.0 planning, but they are blockers to treating v5.9 as a fully stable foundation.

## 5. Major Findings

### F1. Framework-first sequencing is scientifically correct

Severity: Positive finding  
Area: program direction

The plan correctly prioritizes enabling frameworks before adding chemistry. Calibration, uncertainty propagation, and lifetime modeling are the correct infrastructure for moving from a semi-quantitative demonstration simulator toward a realistic process-design tool.

This is consistent with wet-lab reality. HIC, lectin affinity, IMAC leaching, and Protein A lifetime cannot be predicted reliably from reagent identity alone. They require measured resin behavior, target-protein data, operating conditions, and confidence intervals.

Recommendation:

Keep the framework-first strategy, but harden v5.9 routing and contract behavior first. Otherwise v6.0 frameworks will sit on top of unverified process routing.

### F2. Calibration framework is must-have, but the plan under-specifies calibration semantics

Severity: High  
Area: scientific validity, user trust

The plan is correct that a calibration framework is the highest-leverage v6.0 feature. However, the statement that the framework has "no data requirements" is only true for software scaffolding, not for scientific usefulness.

Calibration entries must be more than parameter-name/value pairs. They need validity domains:

- target molecule identity,
- profile key and ligand chemistry,
- column geometry,
- bead batch or material batch,
- temperature,
- pH,
- conductivity or salt type,
- gradient conditions,
- flow rate/residence time,
- units and unit conversion,
- measurement type, such as static binding capacity, DBC10, DBC5, q_max, K_L, K_eq, or m_salt,
- confidence and experimental replicate count.

Recommendation:

Define calibration as a typed schema with validation and units. Calibration should never silently override defaults. The UI and logs should show exactly which values were overridden, from what source, and whether the calibration applies to the selected process state.

### F3. New `src/emulsim/uncertainty/` package conflicts with existing `src/emulsim/uncertainty.py`

Severity: Critical  
Area: implementation reliability

The v6.0 plan proposes a new package:

```text
src/emulsim/uncertainty/
```

But the current repository already contains:

```text
src/emulsim/uncertainty.py
```

Python cannot cleanly support a module file and package directory with the same import path in the same package. This will create import ambiguity and likely break existing CLI or code paths that import `emulsim.uncertainty`.

Recommendation:

Do not create `src/emulsim/uncertainty/` under that name unless the existing module is deliberately migrated. Use one of these approaches:

- Rename the new package to `uncertainty_propagation/`.
- Migrate `uncertainty.py` into a package in a dedicated compatibility change.
- Keep `uncertainty.py` and add new modules such as `uncertainty_m2.py` and `uncertainty_m3.py`.

This must be resolved before WN-F2.

### F4. M1 uncertainty propagation is necessary, but default CVs must be labeled as assumptions

Severity: High  
Area: uncertainty, false precision

The plan correctly identifies false precision as a major issue. However, using literature-typical CVs as defaults can also create false confidence if presented as measured uncertainty.

Real M1 uncertainties are correlated. Bead size, pore size, porosity, ACS density, and mechanical properties are not independent random variables. A Monte Carlo wrapper that samples each field independently may produce plausible-looking but physically inconsistent microsphere states.

Recommendation:

Implement uncertainty in two tiers:

- Tier 1: user-supplied measured uncertainty, with correlations if available.
- Tier 2: assumed uncertainty, clearly labeled as screening-only.

Add sanity constraints so sampled M1 states remain physically valid. Preserve correlations where M1 can provide them.

### F5. Existing `M1ExportContract` already has pore-size standard deviation but not a full uncertainty contract

Severity: Medium  
Area: interface design

The current `M1ExportContract` includes `pore_size_std`, but no formal `M1UncertaintyContract`. This is useful because v6.0 does not have to start from zero for pore variability. However, `pore_size_std` is not enough for full uncertainty propagation because it does not describe bead-size variation, ACS-density variation, porosity uncertainty, or correlation structure.

Recommendation:

Leverage existing `pore_size_std` immediately for accessibility warnings and optional log-normal PSD approximations, but add a formal uncertainty object for full Monte Carlo.

### F6. Lifetime modeling is useful, but must not be over-positioned as universally must-have

Severity: Medium-High  
Area: process economics, scientific framing

The plan calls lifetime modeling must-have because industrial users care about column lifetime. That is true for downstream process economics, especially Protein A and IMAC. But fresh-media performance is still a legitimate and useful simulation target. Lifetime should not block all v6.0 chemistry work if calibration and routing are not yet stable.

Scientific caveats:

- Protein A capacity decay is not only exponential activity loss.
- CIP conditions, feed impurity profile, storage, sanitization, and fouling dominate real lifetime.
- Literature leaching values are resin- and assay-specific.
- IMAC metal leaching depends on metal ion, chelator, buffer, pH, reducers, imidazole, and regeneration protocol.

Recommendation:

Implement lifetime as a calibrated empirical projection module. Mark default lifetime constants as illustrative. Avoid reporting lifetime pass/fail decisions unless calibrated to the resin and process.

### F7. HIC isotherm direction is correct, but default salt coefficients are high-risk

Severity: High  
Area: HIC scientific validity

The proposed HIC isotherm:

```text
K_eff = K_0 * exp(m * C_salt)
```

is a pragmatic and common empirical form. It is appropriate for a simulator if calibrated. However, the plan’s default `m_salt` values are protein-dependent and salt-unit dependent. The plan references values in `(mol/kg)^-1`, but the codebase often uses `mol/m^3` or scalar gradient values. A unit mismatch of 1000x would destroy the model.

Recommendation:

Make HIC calibration required for quantitative use. Add strict units:

- salt concentration basis: molal, molar, or mol/m3,
- salt identity,
- temperature,
- protein target,
- ligand type,
- gradient direction.

Default HIC should be `requires_user_calibration` or `ranking_only`, not `mapped_estimated`, unless the user supplies K_0 and m_salt.

### F8. Lectin competition model is scientifically plausible but needs cofactor and oligomer state

Severity: Medium-High  
Area: lectin affinity, M3 realism

The plan correctly notes that lectin elution can be modeled as competitive binding between glycoprotein and free sugar. This is a valid approximation.

Missing real-world factors:

- Con A requires Ca2+/Mn2+ for carbohydrate binding.
- Con A oligomeric state changes with pH.
- WGA binding is multivalent and glycan-pattern dependent.
- Sugar competitor affinity varies strongly with buffer and ligand presentation.
- Immobilization orientation changes apparent affinity.

Recommendation:

Implement lectin models as `requires_user_calibration`. Include cofactor requirements and sugar competitor identity in `process_state`. Use generic competitive affinity only as a structural model, not as a predictive default.

### F9. TMAE via DVS remains chemically weak and should stay deferred

Severity: High  
Area: chemical validity

The plan appropriately defers TMAE to v7.0, but its chemistry description still needs correction. DVS Michael addition with amines forms beta-amino sulfone linkages, not thioether linkages unless the nucleophile is a thiol. TMAE reagent identity is also ambiguous: a quaternary trimethylammonium species is not a conventional neutral amine nucleophile.

Recommendation:

Keep TMAE deferred. Before implementation, specify:

- exact reactive precursor,
- whether it is tertiary amine, amino alcohol, or pre-quaternized reagent,
- expected installed group,
- reaction mechanism,
- kinetic evidence,
- whether the profile is mechanistic or ranking-only.

Q chemistry already covers the strong anion-exchange use case sufficiently for now.

### F10. Current M3 routing baseline must be fixed before HIC and gradient claims

Severity: Critical  
Area: M3 process reliability

The v6.0 plan depends heavily on M3 routing. Current adapter code exists, but direct testing shows the IEX/SMA route fails. The adapter also uses comments suggesting Molar units while `IMACCompetitionIsotherm` and other M3 code document concentrations in `mol/m^3`. The irreversible adsorption adapter uses scalar conditional logic that is not safe for array inputs used by LRM.

Recommendation:

Before v6.0 alpha, add tests for:

- `select_isotherm_from_fmc()` for every `binding_model_hint`,
- SMA route constructor compatibility,
- IMAC route unit consistency,
- irreversible adapter with scalar and NumPy array input,
- adapter use inside `solve_lrm()`,
- adapter use inside `run_breakthrough()`.

This is a baseline hardening step, not a v6.0 feature.

### F11. Multi-component gradient elution is correctly must-have but high-risk

Severity: High  
Area: transport solver, process realism

The plan correctly identifies multi-component gradients as must-have for IMAC, HIC, IEX, lectin, and mixed-mode workflows. Current gradient code supports a scalar `GradientProgram`, and current gradient elution code reads the gradient but does not pass it into equilibrium loading.

Real v6.0 needs a time-varying process state:

- salt concentration for IEX/HIC,
- imidazole for IMAC,
- sugar competitor for lectins,
- pH for Protein A and mixed mode,
- possibly conductivity instead of salt molarity.

Recommendation:

Make `ProcessState` a first-class dataclass rather than a loose dict. Extend gradient support to multiple named components. Require end-to-end tests where changing imidazole or salt gradient changes the chromatogram, not just the displayed gradient line.

### F12. UI overhaul is necessary and should not be deferred behind API workarounds

Severity: High  
Area: usability, reliability

The plan says some UI panels can be postponed because Python API workarounds exist. That is risky. Users are asking this platform to simulate wet-lab and process workflows, and invalid workflow ordering can easily produce misleading outputs.

Current UI exposes `SPACER_ARM` but not all v5.9 step types. It does not expose metal charging, protein pretreatment, washing, calibration, uncertainty, lifetime, or multi-component gradient configuration. Therefore a complete v6.0 backend without UI gating will remain hard to use correctly.

Recommendation:

Make F6a mandatory before public use of any new framework. At minimum, UI must:

- expose all executable step types,
- filter reagents by current state,
- show intermediate ACS state,
- show FMC and M3 support level,
- block invalid metal/IMAC/protein pretreatment workflows,
- display calibration and uncertainty assumptions.

### F13. Calibration UI should be must-have if calibration framework is must-have

Severity: Medium-High  
Area: user workflow

The plan classifies the calibration framework as must-have but calibration UI as nice-to-have. This creates a usability mismatch. If HIC and lectin modes require calibration, users need a reliable way to inspect and supply calibration values.

Recommendation:

Make a minimal calibration UI must-have:

- load JSON,
- show applied overrides,
- show validation errors,
- show unmatched calibration entries.

Manual editing can remain nice-to-have, but import/inspection should be core.

### F14. PSD integration should be limited to M1-supported distributions

Severity: Medium  
Area: pore accessibility

PSD integration is scientifically valuable, but the plan correctly marks it nice-to-have. Current EmulSim targets agarose/chitosan hydrogel microspheres, where mean-pore approximations may be adequate for screening. PSD is most important for broad, bimodal, or controlled-pore materials.

Recommendation:

Use existing `pore_size_std` for a first log-normal approximation, but do not add a full discrete PSD interface unless M1 can produce real bin fractions. Otherwise the UI will invite users to invent distributions that look quantitative but are unvalidated.

### F15. Full pH kinetics are useful, but enabling pKa fields on all profiles may regress calibrated behavior

Severity: Medium  
Area: kinetics, regression risk

The plan proposes populating `pKa_nucleophile` for all profiles. This can improve chemistry but may break previous parameter fitting if existing `k_forward` values were implicitly fit at the operating pH. Multiplying them by ionization fractions can double-count pH effects.

Recommendation:

For each profile, explicitly define whether `k_forward` is:

- pH-normalized intrinsic rate constant, or
- observed rate constant at `ph_optimum`.

Only apply pH scaling to the former, or rescale profiles so `k_eff(ph_optimum) = k_forward`.

## 7. Revised Acceptance Criteria

Add these to the v6.0 acceptance criteria:

1. Existing v5.9 adapter routing is fixed and tested before v6.0 framework work begins.
2. `select_isotherm_from_fmc()` works for `charge_exchange`, `metal_chelation`, `salt_promoted`, `near_irreversible`, `fc_affinity`, `mixed_mode`, and `lectin_*`.
3. Unit conventions for salt, imidazole, sugar, and pH are documented and tested.
4. `src/emulsim/uncertainty.py` conflict is resolved before adding any `uncertainty` package.
5. Calibration entries include parameter units, target molecule, process conditions, validity domain, and source.
6. Calibration overrides are visible in FMC and in UI.
7. HIC default parameters are labeled `requires_user_calibration` unless user data are supplied.
8. Multi-component gradient process state actually changes equilibrium loading during LRM integration.
9. IMAC imidazole elution test shows binding at low imidazole and elution at high imidazole using the same transport pathway that UI calls.
10. Lifetime projections are labeled empirical and include the assumed CIP/storage/feed conditions.
11. Monte Carlo uncertainty supports default-CV screening and measured-CV modes separately.
12. UI exposes all 9 step types and all executable profiles, including v5.9 metal charging, pretreatment, and washing.
13. UI displays calibration, uncertainty, M3 support level, and confidence tier next to final results.
14. All new package/module names avoid import-path conflicts.

## 8. Recommended Development Sequence

Recommended corrected sequence:

1. v5.9 hardening: fix M3 adapter routing, unit conventions, and array safety.
2. v5.9 hardening: expose all existing 9 M2 step types in UI or explicitly mark hidden API-only functions.
3. v6.0-alpha: implement calibration framework with schema validation and override audit trail.
4. v6.0-alpha: resolve uncertainty module naming and add M1 uncertainty contract.
5. v6.0-alpha: implement lifetime model as empirical calibrated projection.
6. v6.0-beta: implement multi-component process state and gradient-aware LRM.
7. v6.0-beta: implement HIC isotherm with calibration-first defaults.
8. v6.0-beta: implement IMAC imidazole gradient end-to-end validation.
9. v6.0-rc: implement UI panels for workflow, calibration, gradient, uncertainty, and lifetime.
10. v6.0 full or v6.1: add lectin competition and PSD integration if stable.

This sequence reduces the risk of adding models that are chemically meaningful but not actually connected to the process simulator.

## 9. Final Reliability Assessment

Current M2 backend reliability:

- Good for implemented v5.9-style ACS and workflow tests.
- Good profile coverage for semi-quantitative chemistry exploration.
- Still limited by UI exposure and by M3 process integration.

Current M3 reliability:

- Existing generic M3 tests pass.
- Known LRM mass-balance warning cases remain.
- Chemistry-specific adapter routing is not fully tested and at least one route currently fails.
- Gradient elution does not yet use gradient values to change binding equilibrium.

v6.0 plan reliability as written:

- Strong scientific direction.
- Good recognition of calibration and uncertainty needs.
- Over-optimistic about the stability of the v5.9 baseline.
- Under-specifies calibration domains and unit handling.
- Underestimates UI and M3 process-state integration effort.

v6.0 plan reliability after correction:

- Scientifically defensible as a semi-quantitative, calibration-enabled simulator.
- Suitable for process exploration and wet-lab planning when calibrated.
- Not a substitute for measured resin characterization, breakthrough experiments, or regulatory validation.

## 10. Final Recommendation

Proceed with v6.0 planning, but add a v5.9 hardening gate before implementation. The v6.0 framework-first strategy is the right direction, but the baseline must be made technically real: M3 routing must work, units must be consistent, UI must expose existing chemistry, and import-path conflicts must be resolved.

The highest-priority corrections are:

1. Fix and test M3 `binding_model_hint` routing before adding HIC or lectin models.
2. Rename or migrate the existing `uncertainty.py` before creating an `uncertainty/` package.
3. Make calibration schema validation and override transparency mandatory.
4. Treat HIC, lectin, lifetime, and uncertainty outputs as calibrated or screening-only, never as default quantitative predictions.
5. Make multi-component process state a first-class object for gradients.
6. Make minimal calibration and workflow UI part of the v6.0 must-have set.

With these changes, v6.0 can become a credible transition from a semi-quantitative chemistry simulator to a calibration-enabled M2/M3 process simulation system.
